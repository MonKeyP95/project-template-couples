# DECISIONS.md

## Important Decisions & Why

| Decision | Why | Date |
|---|---|---|
| **A trip category can be elaborated with describe-only detail tags via a `details text[]` on `expense_categories`, not a new table or a separate per-trip list** | The category row already survives rename and cascades on delete, and the Profile is a second access point to the same shared categories (per the "reuse existing systems" rule), so hanging details off the existing row keeps one source of truth and zero new joins. Details are profile-intent only — they never touch money (spend still keys on category name) — so a plain `text[]` with last-write-wins is right for a two-person app; no per-item RPC or optimistic locking. Writes live (whole-array replace + `router.refresh()`), matching add/remove category, independent of the wizard's final Save. AI does not read details yet — that's the profile-aware-suggestions slice. | 2026-07-12 |
| **The trip right rail's 7-day weather bar shows the destination's real next-7-days Open-Meteo forecast, not a seasonal estimate around the trip dates; the single-day `getWeather` stub stays** | Asked which 7 days to show, the user chose the literal next 7 days. That only carries information with a *real* forecast — the existing stub varies only by latitude and month, so seven consecutive days come back nearly identical (a flat, useless bar). Open-Meteo's daily endpoint is free, keyless, and returns exactly high/low/WMO-code per day, so it's a single `fetch` with no new dep or vendor. Scoped to the desktop right rail (the literal "top right") and added as a *new* `getWeekForecast`/`getTripWeekForecast` pair rather than rewriting `getWeather` — `/on-the-road` still wants the single current-conditions card, so the stub is left in place. Fixture-driven bar (`detail.weather`, demo-slug-only) replaced. Deferred: a real forecast only reaches ~16 days out, so trips further away would need a seasonal-estimate fallback — not built, since "next 7 days" is what was asked for. | 2026-07-12 |
| **Suggest and the `⌕` door stay separate engines (introspective vs. outward) — no merge this slice** | Suggest reads the trip's own data within a chosen scope (page/trip/day/free-text) for one focused piece of advice — no web search; the door runs outward web discovery for real places with citations + the learning corpus. Different mechanics and output shapes, so one blind merged call would be worse at both and would put "find a place" on surfaces (budget/packing) where it's nonsense. The scope picker is the shared interaction *pattern*, not a shared engine (`generateSuggestion` and the discovery route stay independent seams). Revisit only if the two front doors confuse users in real use. | 2026-07-10 |
| **Overview over-eagerness handled by a prompt restraint line now; a density/restraint preference is deferred** | A one-shot suggestion already returns a single card, so an overview can't march through the whole day in one press — over-eagerness only shows across repeated "another" presses. A cheap restraint line in the trip/day prompts ("surface the single most valuable thing; don't assume every day needs a dinner or every gap needs filling — some open time is intentional") blunts it in v1. The real fix — a couple-level or learned density preference (how much planned vs. left open) consumed by the prompts — is deferred to the behavior-harness slice (vision #2). | 2026-07-10 |
| **One AI access point: a single inline collapsible `assistant` block whose expand/collapse IS the AI on/off; the floating pill, the standalone toggle, and the per-surface suggestion cards + discovery doors are folded into it** | Three stacked AI surfaces (floating pill+chat, per-surface `AiSuggestion` cards, the discovery doors) became one labelled block per page: the header is the on/off — collapsed = off (just the `assistant` label, nothing renders, no model call), expanded = on (`/ suggest` + a bare press-to-open door + inline chat). State persists per-person via the existing `ai` cookie / `useAiMode` (default collapsed = off), so it follows you across pages. This **relaxes** the 2026-06-16 "everything AI hidden when off" rule to "one `assistant` label is always visible as the entry point; nothing else appears and no Claude call happens until you expand." Removed: the floating `assistant.tsx` pill + chat panel, `AiToggle` (including the `/profile` settings row — the block header is now the single switch), and `ai-suggestion.tsx` (its suggest logic moved into the block). The **bare door**: an unlabelled `⌕` line → press → a vertical category list → pick one → that category's existing `DiscoverySection` (press-only, mobile-first; list not chips; `RoadPlaceDoor`/`PlanningPlaceDoor` feed a shared `PlaceDoor`). Budget drafter left separate — revisit its `useAiMode` coupling (it now shows when the block is expanded on the budget tab). Suggest-only invariant unchanged. | 2026-07-09 |
| **Budget planner goes real by drafting only the *content*, not the interview structure: `draftBudgetSeeds` fills the deterministic `planBudgetSteps` scaffold; a forced `propose_budget` tool with no `web_search`; failure falls back to the scaffold with a quiet note** | The five-category/location-grouping structure is UI, not intelligence, so keeping it deterministic shrinks the model's job to "fill these buckets", preserves the working grouping invariant, and lets the interview always open. `draftBudget` (server action) merges Claude's flat items into buckets by matching the echoed location *name* to a group (ids are error-prone to echo). No `web_search`: a budget is an estimate, and web search tripled discovery latency (2026-07-07 row) for no gain — Claude's parametric cost knowledge answers in one round-trip (a live Morocco draft returned 20 well-formed, budget-brief-aware items). On any model failure the action returns the mock scaffold with `drafted:false` and the drafter shows "using rough estimates" — honest, never stuck. Edit-budget (saved rows exist) makes no AI call. Live web-grounded prices deferred. | 2026-07-09 |
| **Chat is the first mock-to-real AI swap: real `claude-sonnet-5` behind a server action, with its own `CHAT_MODEL` constant and a provider-neutral seam signature** | Honors "AI provider is one file" — `chatReply` lives in `claude.ts`, takes the app's own `ChatMessage[]` + a plain context string and returns a string, so swapping to a cheaper model is one constant and swapping provider is a one-function body rewrite (no abstraction built now, per YAGNI). Chat gets its own model constant so it can drop to Haiku without disturbing discovery's web-search flow, which wants a stronger model. The call moved server-side (the key can't ship to the browser) via a `"use server"` action that also builds basic trip context and the dates-driven planning/on-the-road hint. Not gated by the AI toggle — chat always works; the toggle only gates the proactive surfaces. | 2026-07-08 |
| **Learning layer = two stores: an append-only `event_ratings` corpus + an editable per-category `couple_summaries` markdown summary; the summary is regenerated by *evolving* the current text (hand-edits included), and staleness is a percentage of the corpus (≥20% new, floor 3), not a fixed N** | Raw data and its distillation are two different jobs, so they're two tables: the corpus is durable (workspace-anchored, survives event/day/trip edits and the AI toggle), the summary is the thing the couple reads/edits and the AI consumes. Feeding the current summary back as the base to evolve means a hand-edit is never lost and auto-replace is always safe — which removes the need for any "your edit will be overwritten" nudge. Percentage staleness scales the bar to corpus size: at 5 ratings one more is 20% and worth a refresh, at 30 it's 3% and isn't. `saveCoupleSummary` deliberately omits `rating_count_at_generation` so a manual edit doesn't clear staleness (the next load still regenerates and folds the edit in). Best-effort `category` tag (meal-word heuristic) because per-category % needs per-category counts; the summariser still reads the raw text, so a mis-tag is low-stakes. | 2026-07-08 |
| **Learning "position 3": logging is AI-free, generation is AI-gated and lazy on profile load, the toggle itself never generates, and discovery reads the summary but never generates it** | Keeps signal capture independent of the AI toggle (you never lose ratings by having AI off) while keeping every Claude call behind `isAiEnabled()`. Flipping AI on has no side effect; the work happens where the summary lives — the next couple-profile load with a stale category background-regenerates (non-blocking, ref-guarded once). Discovery stays fast and read-only (it falls back to the structured prefs when no summary exists), so generation lives in exactly one place. Trade-off accepted: turn AI on and go straight to a search without visiting the profile, and that first search has no learned block yet. | 2026-07-08 |
| **Restaurant discovery uses the basic `web_search_20250305` tool, not the newer `web_search_20260209`** | The `_20260209` variant has built-in "dynamic filtering" that runs server-side `code_execution` to pre-filter search results — a token/accuracy optimisation that isn't worth its latency cost for "restaurants near X". A timed A/B (same bare query, model, `max_uses:3`) measured `_20260209` at 176.7s (code_execution ran) vs `_20250305` at 35.8s, both returning 4 valid suggestions — a ~5x speedup for no quality loss. Revisit if a future feature genuinely needs the pre-filtering. | 2026-07-07 |
| **Shareable trips: one read-only `security definer` projection RPC (`shared_trip(token)`) is the entire privacy boundary; copy runs under the caller's own RLS (no privileged write)** | The shareable content (itinerary skeleton) is cleanly separable from sensitive tables (expenses/budget/savings/members), so the projection just never selects them — safe by construction, not by remembering to hide fields. Anonymous view needs only `grant execute … to anon`; base tables stay closed. The copier already has full insert rights in their own workspace, so cloning is plain TS sourced from the projection — no second security-definer function. The `/t/` route is allow-listed in `proxy.ts` so logged-out visitors aren't bounced to `/signin`. | 2026-06-25 |
| **Budget planning lives on the itinerary spine (per-location + trip-wide), not a standalone editor; located items inherit location dates** | First slice of the planning-spine vision. Budget items already carry `location_id`, so weaving cost into the itinerary is mostly presentation. Located costs inherit their location's dates (no per-item picker); only trip-wide costs carry `when_start`/`when_end`. A scoped save keeps other locations untouched. The AI-off standalone editor is retired; the drafter stays until a later slice. | 2026-06-17 |
| **Budget line items are server-persisted (`trip_budget_items`, RLS), shared; planned total = sum of items; editable in both AI modes** | Supersedes the earlier "total-only, no per-item persistence" decision. The user wanted a real, shared, viewable/editable list of hotels/prices, including in AI-off mode. A replace-all `saveBudgetItems` keeps `trips.planned_budget_cents` = sum, so existing budget views are unchanged. The AI drafter now loads/saves the same table instead of localStorage. | 2026-06-17 |
| **AI mode is a per-person cookie (`ai`), off by default; one global switch gates all AI surfaces; suggest-only** | The user wants AI off unless explicitly turned on, and constrained even when on. A cookie mirrors the theme/timezone prefs (no schema); a root-layout context provider gates suggestion cards, trip chat, and the budget assistant via `useAiMode()`. Off swaps the budget drafter for a manual total field so budgeting still works. The suggest-only invariant (code under `lib/ai` never writes) keeps a future real model from acting autonomously. | 2026-06-17 |
| **Budget assistant is a hybrid: Accommodation + Activities grouped by location, other categories flat** | Brings back the location-by-location "walk through your places" smoothness (each place a sub-group, multiple hotels/activities per place, location-seeded) without losing the category correctness — trip-wide costs (transport/food/other) stay flat. Rows key per bucket (`step.key` or `step.key:locationId`); still a total-only mock behind `planBudgetSteps`. Earlier the assistant had gone fully category-centric; this is the deliberate re-balance. | 2026-06-16 |
| **Mock AI suggestions behind `suggestionFor(surface)` (`lib/ai/suggestions.ts`), rendered via the shared `SuggestionCard`** (now with optional `onApply`/`onDismiss`); shown on budget/trip/notes/home/road/checklists | Extends the existing static SuggestionCard placeholder to the rest of the app without a key/cost. Dismissal is session-only (per device); real model is a one-function swap (`suggestionFor` -> async LLM call). Existing Packing/Itinerary cards left untouched (pass no callbacks, stay inert). | 2026-06-15 |
| **Trip chat built mock-first behind `requestChatReply` (`lib/ai/chat.ts`); mounted in a trip-segment `layout.tsx`** | An open chat can't be usefully mocked in content, but the surface + seam can be built without a key/cost. Replies are an honest placeholder now; the real model is a one-function swap (`requestChatReply` -> `fetch('/api/chat')`). The layout mounts it so the conversation survives `?tab=` navigations (tabs are URL nav, not client state). Provider/key/route/history deferred. | 2026-06-15 |
| **AI assistant built mock-first behind a pure `planBudgetSteps(input)` seam** (`lib/ai/budget-planner.ts`); no Anthropic SDK, no new tables; the first feature is a guided budget interview that writes via the existing `updateTripBudget` | Lets us build and use the whole interaction surface now and validate the seam without API cost or keys. The seam returns an interview (steps + suggestions), not a number, so real Claude is a one-file swap: make `planBudgetSteps` async, generate the steps from the LLM client, add `await` at call sites. | 2026-06-15 |
| **Budget assistant is planning-only and category-centric: total-only, no per-category/per-item persistence, no planned-vs-actual** | Matches the Budget(plan)/Expense(reality) split — planning helps you build a number, comparison-to-actual is stats and lives elsewhere (or nowhere yet). The interview walks categories (Accommodation/Transport/Food/Activities/Other), each an add-list of `subject / when / cost` items; money rolls up to one trip total. (An earlier location-by-location walkthrough was dropped in favour of category steps with a per-item `when` field — the user wanted detailed line items like "Sea hotel · 3 days · 350".) | 2026-06-15 |
| **Budget tab reorganized into a `Budget` / `Saved` / `Settle up` pill bar** (matching packing); `SegBtn` lifted into the shared `together` library with a `tone` prop; `BudgetFigures` split into `SpentFigure` + `SavedFigure` | One long scroll mixed three concerns; segmenting matches packing's mental model and lets each figure render under its own pill. Lifting `SegBtn` avoids a copy-paste between the two tabs. Pure presentation — no data/query/migration change. | 2026-06-15 |
| **Budget pill split into `Budget` (planning) + `Expense` (tracking)**; compact settle strip added to both via an extracted `SettleUpButtons`; settlement history added to the `Settle up` pill | Separates "what you'll spend" (figure + budget-by-location) from "what you spent" (add expense + ledger). Settling is reachable where you work without leaving the view, while the full breakdown + history stay in one place. History derives from `is_settlement` expense rows, so no new data. Pure presentation. | 2026-06-15 |
| **Budget figures moved out of the pills header into each view's content** so the pills bar stands on its own (dusk-tint header = label + pills only) | The header had crowded the label, pills, and per-view figure together; pulling the figure into the content makes the pills a clean, standalone bar and reads as the first row of each view. Pure presentation. | 2026-06-15 |
| **Add-expense + ledger shown in the Budget pill too** (in addition to Expense) | User wants to log and review expenses without switching to the Expense pill; the two components are independent client instances, so duplicating them across views is safe. Pure presentation. | 2026-06-15 |
