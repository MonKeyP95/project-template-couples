# DECISIONS.md

## Important Decisions & Why

| Decision | Why | Date |
|---|---|---|
| Position as a **couples/families travel app**, not a generic household organizer | Crowded space (Cozi/FamCal) when generic; clear differentiator when trip-centric. Personal use case is travel. | 2026-05-25 |
| Product name: **Together** | Picked one name and stuck with it; previously inconsistent ("LifeTogether" vs. "Together"). | 2026-05-25 |
| Model shared surface as **workspace with members + roles**, not strict 2-person "couple" | Same schema scales from couple → family → small group without rewrite. MVP just hides the multi-member UI. | 2026-05-25 |
| **Drop Drizzle from MVP**, use `supabase-js` directly | Drizzle + Supabase RLS is friction-heavy. `supabase-js` is enough until queries get complex. Revisit at Phase 4+. | 2026-05-25 |
| **One AI provider** (Anthropic Claude) for MVP, not a provider-agnostic abstraction | Premature abstraction before any AI feature exists. Wrap calls in one module so swapping later is cheap. | 2026-05-25 |
| **Shadcn/ui on day one** | Standard pairing with Next.js + Tailwind in 2026; deferring it just rewrites components later. | 2026-05-25 |
| Next.js 16 + TypeScript + Supabase | Beginner-friendly, fast to iterate, good fit for AI features and real-time shared data. Scaffolder pulled Next 16 (newer than the Next 15 originally planned for); accepted as the current stable. | 2026-05-25 |
| **pnpm** as package manager, **Tailwind v4** (PostCSS plugin), **`src/` directory**, **Turbopack** dev bundler | All defaults of `create-next-app@latest` in May 2026. Accepted rather than overriding — they're the path of least surprise for future contributors and current docs. | 2026-05-25 |
| **Approve build scripts for `sharp` + `unrs-resolver`** in `pnpm-workspace.yaml` | pnpm 11 requires explicit opt-in for postinstall scripts. `sharp` powers `next/image` optimization; `unrs-resolver` is used by ESLint. Both are first-party deps of the scaffold. | 2026-05-25 |
| Shadcn style: **`base-nova`** with **`neutral`** base color (current `create-next-app` defaults) | Accepted defaults to get unblocked. Palette will be re-themed to the warm pink/teal in `DESIGN.md` at task 5 (landing page). | 2026-05-25 |
| Shadcn primitives: **`@base-ui/react`**, not Radix | shadcn 4.x ships components on `@base-ui/react`. Don't add `@radix-ui/*` packages; it's no longer the underlying layer. | 2026-05-25 |
| **`shadcn` is a runtime dep**, not just a CLI | shadcn 4.x ships `tailwind.css` via `node_modules/shadcn/dist/tailwind.css` which `globals.css` imports. Removing it from `dependencies` would break the theme. Leave it. | 2026-05-25 |
| **Disable build script for `msw`** in `pnpm-workspace.yaml` | `msw` is a transitive dep of `shadcn`'s registry tooling, not used at app runtime. Its postinstall sets up a service worker we don't need. | 2026-05-25 |
| Env var naming: **`NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`** (new Supabase convention), not `..._ANON_KEY` | Supabase renamed the public key to "publishable" in late 2025. Using the new name forward — even though our project issued a legacy `anon`-role JWT, the value works in either env var name. | 2026-05-25 |
| Use **`proxy.ts`** (Next.js 16 convention), not `middleware.ts` | Next 16 renamed the middleware file convention to "proxy." `middleware.ts` still works but emits a deprecation warning. Confirmed via Next 16 upgrade docs. **Side effect**: proxy runs in the **Node.js runtime**, not Edge — fine for Supabase session refresh, slightly different perf profile from the old Edge middleware. | 2026-05-25 |
| `@supabase/ssr` cookie pattern: **`getAll` / `setAll`**, not per-cookie `get/set/remove` | The post-0.5 API. Pre-0.5 individual-cookie methods are deprecated. Pattern is identical across `src/lib/supabase/server.ts` and `src/proxy.ts` — the only difference is where the cookies live (request vs. `cookies()`). | 2026-05-25 |
| Fonts: **Instrument Serif** (display, italic) + **Hanken Grotesk** (body) + **JetBrains Mono** (small marks) | Explicit avoidance of Inter/Geist defaults. Serif headlines + warm sans body fit the "calm editorial" direction in `DESIGN.md`. Italic Instrument Serif is the visual hook (the pink "together" word). | 2026-05-25 |
| Style the `Link` with `buttonVariants(...)` instead of `<Button asChild>` | The new shadcn `base-nova` Button (built on `@base-ui/react`) has no Slot/`asChild` prop. The official pattern with this primitive is to apply `buttonVariants` to whatever element you want to look like a button. | 2026-05-25 |
| Landing palette: **OKLCH** tokens, with hand-tuned values matching the `#faf7f5 / #f8c1d4 / #67e8f9 / #374151` palette in `DESIGN.md` | Shadcn's `base-nova` defaults to OKLCH; staying in OKLCH keeps the theming consistent with future shadcn components and gives more perceptually-even hue shifts than hex/HSL. Dark mode tokens are warm-toned (mocha base, not neutral black) so the brand still reads. | 2026-05-25 |
| Apply migrations via **Supabase SQL Editor paste**, keeping SQL files in `supabase/migrations/` for source-of-truth | Avoids installing the Supabase CLI for one migration. Files are still versioned + reviewable. Switch to `supabase db push` (CLI) once we have 3+ migrations and the setup cost amortizes. | 2026-05-25 |
| **Dropped pink primary → deep sea**; rebased to a sand-and-sea palette (`sea/sand/clay/moss/dusk` + tints) per `design_handoff_together_app/README.md` | Phase 3 design hi-fi removes pink entirely. Centre of gravity shifts to per-page surface tints (sea for hero, clay for packing, dusk for budget), so the palette is additive: shadcn semantic tokens stay, extended palette is exposed via `@theme inline`. Body font also moved to IBM Plex Sans + IBM Plex Mono (kept Instrument Serif). | 2026-05-26 |
| **Trips schema split: header in DB, itinerary still fixture** (step 5 of the design handoff) | The handoff sequences `trips` + `trip_members` (step 5) ahead of `itinerary_days` (step 8). Build a `TripHeader` query against Supabase + keep `TripDetail` (itinerary, weather) as a per-slug fixture. The page composes the two — when step 8 lands, only the fixture's `itinerary` array moves to the DB; component shapes and the route stay put. | 2026-05-26 |
| **Lombok seed via auto-discovery `do $$` block**, separate migration file | The user pastes migrations into the Supabase SQL Editor (per the earlier "no CLI yet" decision). `auth.uid()` is `null` in the SQL Editor, so the seed instead picks the oldest workspace that has a member and inserts there. Idempotent: a second run is a no-op once `(workspace_id, 'lombok')` exists. Kept in its own file (`20260526000002_seed_lombok.sql`) so it doesn't pollute the schema migration. | 2026-05-26 |
| **Reuse `is_workspace_member` for trips RLS; add `is_trip_workspace_member` for trip_members** | The Phase 2 migration already shipped `is_workspace_member(workspace_id)`/`is_workspace_owner(workspace_id)` SECURITY DEFINER helpers. `trips` rows carry `workspace_id` directly, so the existing helper works. `trip_members` needs a 1-hop join through `trips`, so a new helper `is_trip_workspace_member(trip_id)` keeps the policies one-liners and avoids RLS recursion. | 2026-05-26 |
| **Disable Supabase Confirm-email** for Phase 2 MVP | Two trusted users; faster sign-up flow. Re-enable + ship password-reset before any wider exposure. | 2026-05-26 |
| **Two flat queries + JS map** instead of `profiles(display_name)` nested embed on `workspace_members` | PostgREST can't infer the join between `workspace_members` and `profiles` (both reference `auth.users.id` separately, no direct FK between them). Nested embed silently returns null. Two flat queries always work. Verified during local E2E when the invite card was failing to render. | 2026-05-26 |
| **Settle-up as `is_settlement boolean` column on `expenses`**, not a separate `settlements` table or category-tag | Keeps the ledger as one table while letting the balance math be exact: `net = (aPaid-bPaid)/2 + aTransfers - bTransfers` where the transfer terms only count settlement rows. Settlement amount = `\|net\|` (the real cash that changed hands), so the row in the ledger reflects the actual transfer instead of an inflated bookkeeping number. A `category='Settlement'` tag would have worked too but required inserting `diff` (not `diff/2`), which makes the ledger row misleading. Separate `settlements` table is correct long-term — defer until expenses gain enough complexity that splitting helps. | 2026-05-27 |
| **`expenses_insert` RLS allows `paid_by` to be any workspace member**, not just `auth.uid()` | The settle-up Server Action inserts a row paid_by the debtor regardless of which member clicked the button. Restricting paid_by to `auth.uid()` (like `packing_items.added_by`) would force a debtor-only button. Both members in a workspace are trusted, so the wider policy is fine. Same policy will support the eventual "+ log expense" form where you record "Monkey paid for X" while logged in as Giraf. | 2026-05-27 |
| **`settleUp` throws on error**; `togglePackingItem` returns a Result | They have different call sites. `<form action={...}>` wants a `Promise<void>`; throwing is the idiomatic error path and Next renders the error boundary. The packing toggle is called from a client `await` so it returns `{error?}` to let the optimistic-update state revert without throwing in the render path. | 2026-05-27 |
| **Planned budget hardcoded in `fixtures.ts`** alongside itinerary + weather | Same rationale as the itinerary fixture: one trip with a known budget, no need for a `trips.planned_budget_cents` ALTER yet. When step 8 moves itinerary into the DB, fold this in. | 2026-05-27 |
| **`itinerary_days` stores `day_date` only**; ordinal (`d`), day-of-week (`dow`), and short date text are derived at render | One source of truth. If trip start/end shifts, ordinals shift automatically — no backfill needed. `dow`/`date` formatting belongs at the view layer anyway (locale, timezone). A day with no plan is just an absent row, not a row with null fields. | 2026-05-27 |
| **Belt-and-suspenders for Realtime: keep Supabase Realtime + add `RefreshOnVisible` (`router.refresh()` on `visibilitychange` / `focus`)** | First end-to-end mobile test showed partner's checks didn't appear unless the page was reloaded. Root cause is platform-level: mobile browsers (especially iOS Safari) suspend backgrounded WebSocket connections and Supabase doesn't replay missed events. Realtime still gives the "live as you watch" feel; the focus-refetch gives correct state on every app-resume. Cheaper and friendlier than PWA push or a native rewrite. Packing-tab also syncs its `initialItems` prop into local state so the refresh actually reaches the client view. | 2026-05-27 |
| **`useTransition` + return-value Server Action for `+ log expense`**, not `<form action={…}>` | The form must stay expanded after success (so the user can batch-enter several expenses) and must surface inline server errors without unmounting. A plain `<form action>` submit always navigates/refreshes and offers no success branch, so the client can't decide whether to clear-and-keep-focus vs. show an error. `settleUp` keeps its `<form action>` shape because its UX is different (one-shot, page can refresh). | 2026-05-27 |
| **`+ new trip` lives on a dedicated `/trips/new` route, not inline on `/home`** | Inline expansion (the pattern from `+ add packing item` and `+ log expense`) was designed for short forms inside a contextual frame (a category row, a budget tab). `/home`'s `+ new trip` button has no such frame — it sits below a greeting, Upcoming card, and Dream board. A focused page sidesteps layout fights at all three breakpoints (390 / md / lg) and matches the one-shot redirect rhythm of "create then leave." The form component is portable; moving to inline-on-home or a shadcn dialog later only changes the shell — the `createTrip` Server Action stays put. | 2026-05-28 |
| **Graceful-empty budget header instead of a `planned_budget_cents` column on `trips`** | The Phase 3.5 TODO scoped fields to `name / slug / dates / country / optional lat/lng`. Adding a planned-budget column + form field is real schema work for a value most newly-created trips won't have at creation time anyway. `budget-tab.tsx` instead branches on `plannedBudgetCents === 0`: hide the planned-cap `<Bar>`, the `/ €planned` text, and the `"X% of planned"` line. Lombok keeps its €2,800 hardcoded in `fixtures.ts` until a future task moves it to the row. | 2026-05-28 |
| **One-table model for dreams + trips** (dates nullable + `fuzzy_when` text label) | A dream is a destination you'd like to visit — i.e. a trip without firm dates yet. Modelling them as separate tables would double up every FK and RLS policy on `packing_items`, `expenses`, `itinerary_days`. Single-table keeps "promoting a dream" as a date edit (no row copy, no row delete, all child rows carry over). The CHECK constraint `(both null) or (both set and end >= start)` enforces no half-states. `fuzzy_when` stays free-text — sortable-date hints can be added later if the need surfaces. | 2026-05-28 |
| **Tone derived from slug hash, no schema column** | The hardcoded Lombok card had a `tone="sea"` topo bg and a hardcoded vibe MonoBadge (`Surf · Dive · Trek`); user-created trips would have neither without a schema column. `slugToTone(slug)` hashes the slug into sea/clay/moss/sand deterministically — same slug always same tone, no DB roundtrip, no form field, no migration. Lombok is special-cased to `sea` so the existing visual is preserved exactly. If per-trip tone becomes a real felt need, swap to a column read and the call sites don't move. | 2026-05-28 |
| **`/home` four-band layout, hero claim rule** | Bands: Hero (one card, soonest active trip) → Trips (other dated, non-hero) → Dreams (undated) → Past (dimmed compact rows). The hero claim rule: earliest `now` trip beats soonest `upcoming` trip for the hero slot. Dreams never claim hero. No separate "Now" section header — the in-card `// now` MonoBadge + the section row reading `Now · 1   day 3 / 8` carry the signal at two visual levels. Past trips stay visible (not hidden) so the day-after-end-date doesn't yank a trip off `/home` for users mid-extension. | 2026-05-28 |
| **Dream toggle in the existing form, not a separate `/trips/new-dream` route** | The form bodies are 90% the same — only the date inputs swap for the `When?` text input. A toggle is one extra control; a separate route would be a duplicate component, duplicate Server Action call site, duplicate validation tree. The Server Action branches on `isDream` and rejects mixed-state submits (`Dreams have no dates.` / `Trips don't have a 'when?' label.`) so the data invariant holds even if the client form is bypassed. | 2026-05-28 |
| **`/trips/[slug]` branches internally for the dream variant**, not a separate `/dreams/[slug]` route | Same reasoning as the form — 90% the same UI. Branch points are isolated to three seams: hero date-range row swaps for `fuzzy_when`; weather strip + desktop weather grid hide; itinerary tab swaps `TabStub` for `DreamItineraryStub`. Packing + budget tabs work for dreams unchanged. One route, three small conditionals; the alternative was a parallel route + parallel page component for negligible visual benefit. | 2026-05-28 |
| **Itinerary tab stays empty for dreams; don't relax `itinerary_days.day_date NOT NULL`** | Relaxing the constraint would require deciding what an "undated itinerary day" even means (and the ordering, the rendering, the ON CONFLICT semantics). A parallel "ideas" sub-table is the other clean option but doubles the schema. The empty stub ("Add dates to plan day-by-day") is honest about the constraint and points the user to the right next step. Revisit when the empty-state actually annoys someone. | 2026-05-28 |
| Build iteratively in small steps | Learning project — cleaner code, easier to debug, validates assumptions early. | 2026-05 |
| Warm & calm visual design | Trip planning should reduce stress, not add to it. | 2026-05 |
